## V1.5.2 - 2026-06-24

### 背景

在网络延迟较高时，快捷 +1、手动记账或设置保存可能需要较长时间才返回结果。此前部分按钮缺少即时反馈，用户容易在等待期间连续点击；快捷 +1 的动画还会在服务端确认前出现，可能造成操作已经完成的错觉。

### 前端防重复提交

* 群组详情页增加同步引用锁，不依赖 React 状态完成下一次渲染后才阻止第二次提交。
* 快捷 +1 按成员 ID 分别维护操作锁和加载状态。同一成员等待响应时无法再次提交，但其他成员仍可正常操作。
* 手动新增、核销、调平、添加成员和修改成员昵称共用详情操作锁。
* 群组基础设置、认领设置、计息设置、认领审核、抵扣额度切换及删除群组共用设置操作锁。
* 操作按钮增加 `disabled`、`aria-busy`、等待光标和“处理中/保存中”文案；快捷 +1 加载图标兼容减少动画偏好。
* 快捷 +1 的浮动成功动画移动到接口成功响应之后，失败请求不再展示成功反馈。

### 服务端幂等保护

* 新版客户端为 `quickAddRecord` 和 `submitRecord` 的每次操作生成唯一 `operationId`。
* 服务端校验操作 ID 格式，并以“用户 ID + 操作 ID”作为 `OperationReceipts` 凭据文档键。
* 幂等凭据保存请求指纹；如果同一个操作 ID 被用于不同群组、成员、操作类型、数量、余额模式或备注，服务端拒绝执行，防止错误复用。
* 当相同请求被再次送达时，服务端识别已有凭据并返回成功，不再重复修改余额或新增流水。
* 为兼容部署期间仍处于打开状态的旧版页面，缺少 `operationId` 的旧请求由服务端生成临时 ID，不会因前后端版本短暂不一致而直接失败。

### 数据一致性

* 快捷 +1 和手动记账均在同一个 Firestore 事务内写入成员余额、`Records` 流水、`AuditLogs` 管理日志及 `OperationReceipts` 幂等凭据。
* 任一写入失败时事务整体失败，避免出现余额已变化但流水或管理日志缺失的部分成功状态。
* 幂等凭据包含 30 天后的 `expiresAt` 字段，为后续配置 Firestore TTL 清理策略预留条件。

### 影响与取舍

* 同一成员的快捷 +1 在前一次请求完成前不能连续操作；需要一次增加多笔时可使用手动“记一笔”输入数量。
* 每次账务操作会额外写入一条幂等凭据，换取重复请求保护和更完整的数据一致性。
* 前端操作锁主要防止连续点击，服务端幂等保护负责处理同一请求在网络或基础设施层被重复送达的情况。

### 版本与验证

* `package.json`、页面版本标记、应用内更新日志和公开 `CHANGELOG.md` 更新至 V1.5.2。
* `npm run lint` 通过，零错误、零警告。
* `npm run typecheck` 通过。
* `npm run build` 通过，Next.js 16.2.9 生产构建完成。
* `git diff --check` 通过。

## V1.5.0 - 2026-06-23

### 抵扣额度数据模型

* 群组新增 `creditBalanceStatus`，成员负余额解释为抵扣额度；所有用户界面通过共享格式化逻辑将欠款与抵扣额度分开展示。
